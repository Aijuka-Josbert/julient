## рЯФє 4. What to Look Out For (Potential Pitfalls)

| Pitfall | Solution |
| :--- | :--- |
| **DNS propagation** | Can take up to 48 hours, but usually 15вАУ30 minutes. Use `https://whatsmydns.net` to check propagation. |
| **HTTPS not enabled** | GitHub Pages will automatically issue a Let's Encrypt certificate. If not, wait a few minutes and refresh the Pages settings. |
| **Mixed content (HTTP images)** | All internal links are relative, so they'll use HTTPS automatically. If you used external HTTP resources, change to HTTPS. |
| **Internal links with `.html`** | They still work, but if you want clean URLs, keep them as they are вАУ GitHub Pages serves both. |
| **EmailJS keys** | Your EmailJS public key is already in `index.html`. Ensure it's the correct one (it is). |
| **Images not loading** | Check case sensitivity: `labourlaw.jpg` vs `labourLaw.jpg` вАУ Linux is case-sensitive. Your file names are correct. |
| **404 pages** | You can create a `404.html` in the root вАУ GitHub Pages will serve it for missing pages. |

## рЯФє 6. What About `.htaccess` Features?

Since GitHub Pages doesn't support `.htaccess`:
- **Clean URLs (hide `.html`)** вАУ not possible, but you can keep `.html` in URLs вАУ it's fine.
- **Caching headers** вАУ GitHub Pages sets its own good defaults.
- **Security headers** вАУ you can add them via `<meta>` tags in HTML (you already have many) or use GitHub Pages' built-in security (it provides HTTPS, HSTS, etc.).
- **GZIP compression** вАУ GitHub Pages compresses assets automatically.

So **delete `.htaccess`** and don't worry about it.
